The interface health page displays the admin/oper state and error counter samples for every physical interface (`l1PhysIf`) tracked across all fabric nodes. It is useful for spotting flapping links, incrementing error counters, and ports that are operationally down despite being admin-up.

## Columns

| Column | Description |
|---|---|
| Node | Leaf or spine node ID |
| Interface | Interface name (e.g. `eth1/1`) |
| Description | Interface description as configured on the node |
| Admin | Administrative state (`up` / `down`) |
| Oper | Operational state (`up` / `down`) |
| Speed | Negotiated link speed |
| Rx err | Received error counter |
| Tx err | Transmitted error counter |
| CRC | CRC error counter |
| Align | Alignment error counter |
| Last link change | When the interface last changed operational state |
| Sampled | Timestamp of the counter sample shown |

## Counter mode

The toolbar has a **Delta / Current** toggle that changes what every counter column shows:

- **Delta** — the change in the counter since the *previous* sample. This is what you want for spotting interfaces that are *actively* accumulating errors right now.
- **Current** — the raw cumulative counter as reported by the node. Useful for absolute values, but dominated by history, so a small recent increase is hard to see.

<Callout type="info" title="Delta vs. reset">
A **delta** is the increase since the last sample. It can be `Reset` (shown in place of a number) when the raw counter went *backwards* between samples — a device reboot, interface flap, or a counter clear. `Reset` is not zero: it means "we can't compute a trustworthy change here," so that interval is skipped rather than counted as `0`.
</Callout>

## Reading error counts

In **Delta** mode a counter cell tells you three things at a glance:

- **Bold, coloured number** — the counter increased in the latest interval. This interface is actively erroring.
- **Faint `0`** — no change; the interface is healthy for that counter.
- **`Reset`** — the counter went backwards (see the callout above); the interval is not counted.

## Counting CRC view

The **All / Counting CRC** toggle switches the table to a focused view of interfaces that gained CRC errors over a trailing window. Use the **7d / 30d** toggle (shown only in this view) to pick the window.

In this view the **CRC** column becomes a *windowed total* — the sum of CRC errors that interface accumulated over the selected window — and the list is sorted worst-first. A second line under each total shows the latest interval, so you can tell an interface that is *still* erroring apart from one that spiked earlier and has since gone quiet:

<CrcTableExample />

- **`461` · `+4 last poll`** — accumulated 461 CRC errors over the window and is *still* counting (4 in the most recent sample).
- **`128` · `0 last poll`** — gained 128 earlier in the window but was quiet in the latest sample.
- **`512` · `reset`** — a real windowed total; its counter reset between the last two samples, so the latest interval is unknown.
- **`0` · `0 last poll`** — clean; no CRC increase.

<Callout type="info" title="Why a windowed total?">
The window total is what qualifies an interface for this view, so every row explains *why it's here*. A `0` or `reset` in the latest sample no longer makes a genuinely-affected port look fine — the total still reflects the damage over the window.
</Callout>

Above the table, the **7-Day CRC Error Delta Trend** chart sums CRC deltas across all affected ports over the window. Counter resets contribute nothing to this total (they are excluded, not counted as zero), so it reflects real accumulated errors.

## Per-interface trend

Click any row to open its trend drawer. The chart plots **error deltas** — each **bar is the number of new errors counted since the previous sample**, not a running total. Bars make the discrete, per-interval nature honest: taller means more errors that interval, and no bar means none.

<TrendChartExample />

- **Counter reset** — a dashed vertical rule marks where the counter went backwards. The bar before it and after it are not directly comparable.
- **No data** — a shaded band marks a monitoring gap (missing samples); the bars break across it rather than bridging, so an outage is not mistaken for a quiet period.
- **CRC is shown by default.** Click a series label to show or hide it; the range toggle (24h / 7d / 30d / All) sets the time span.

## Sort order

By default, interfaces use a **natural sort** on the interface name, so `eth1/2` sorts before `eth1/10`, and ports appear in port-number order within a node. Clicking a numeric column header sorts by that counter instead. The Counting CRC view defaults to sorting by windowed CRC total, worst-first.

## Filters

- **APIC host selector** — switch between configured APIC hosts
- **Node filter** — show only interfaces from selected nodes
- **All / Counting CRC** — restrict the table to interfaces that gained CRC errors over the selected 7d/30d window

## Resync

Click **Resync** to pull fresh interface state and counter samples from APIC. You will be prompted for APIC credentials. Each resync appends a new counter sample, so the delta counters reflect the change since the previous resync.
